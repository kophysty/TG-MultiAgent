# Будущие интерфейсы (веб, расширение)

Чтобы потом сделать:

- мини-веб страничку
- десктопный клиент
- Chrome extension

нужен один стабильный источник данных и API.

## Принцип

Будущие интерфейсы должны ходить в наш агент-backend, а не в Notion напрямую, потому что:

- оркестрация, предпочтения и правила живут у нас на сервере
- Notion через MCP или REST остаётся "хранилищем и тулом", а не "мозгом"

### Пример API

- `GET /tasks/today`
- `GET /tasks/upcoming`
- `POST /task`
- `PATCH /task/{id}`
- `DELETE /task/{id}`

UI решает, что показать, но данные получает через backend (скорость, кэш, единая логика).
